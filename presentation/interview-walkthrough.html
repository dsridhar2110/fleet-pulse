01 · What this project is for

The problem, in one breath

Hospitals run MRI and CT scanners that must not fail by surprise — a surprise failure cancels a day of patients and costs roughly $27,000 per day of downtime, while sending an engineer to check a machine costs about $800. So missing a failure is ~100× worse than a wasted visit. The machines constantly write logs — sensor readings, error codes, service notes — but nobody can read a million rows a week. FleetPulse reads them instead, and every Monday hands the service team a ranked list: "these 20 machines are most likely to fail in the next 7 days — go look at them first." It's a miniature of what Siemens Healthineers' Customer Service organisation does with its real installed base (Smart Remote Services / Guardian).

02 · The whole thing, simply

What I built, step by step

1

I generated the data myself — honestly.

There's no public dataset of 500 hospital scanners, so I wrote a simulator: each machine's components age and wear out (Weibull failure curves — the standard "things wear out" math), and before most failures the sensors drift for weeks (helium slowly dropping, vibration rising). 15% of failures give no warning at all, so the problem stays honestly hard.

Why it matters: I can defend every data assumption, because I chose each one to copy a known real-world behaviour.

2

Six tables, one key.

A star schema: fleet_master (the 500 machines) in the middle, and five event tables — telemetry (1M daily sensor readings), error_events (323K error codes), failures (440 — the answer key), maintenance (1,454 visits), tickets (1,610 engineer notes) — all joining on machine_id. Stored as Parquet, the compressed column-based file format every big-data tool speaks.

3

PySpark turned events into features.

For every machine, every Monday, I computed 226 numbers describing "what the last 7/14/30 days looked like": rolling averages, trend slopes, error-burst counts, days since maintenance. Rule number one: a feature may only use data from before that Monday — never after. A unit test (test_no_leakage.py) enforces it. That's leakage prevention — the #1 way ML projects secretly cheat, and mine provably doesn't.

4

XGBoost predicts; calibration makes it honest.

XGBoost (a gradient-boosted tree model — great on tabular data, handles missing values, explainable) outputs each machine's probability of failing within 7 days. I then applied Platt calibration so a score of 0.40 really behaves like a 40% chance — which is what lets me do money-math with it later.

I chose XGBoost over deep learning on purpose: few failure examples, tabular features, and the service team needs to know why a machine is flagged (SHAP explains each prediction).

5

I graded it the honest way.

Failures are only 1.7% of machine-weeks, so "accuracy" is a lie (a model saying "never fails" is 98% accurate and 100% useless). I report PR-AUC (0.48 vs 0.013 random), recall@20 (a weekly top-20 list catches 78% of failures), precision@20 (0.23), and a median 3-day warning. Time-based split: trained on Jan–Aug, tested on Nov–Dec it had never seen.

6

The Business Impact Engine turned scores into dollars.

Expected loss per machine = probability × typical downtime × $27k. Then a backtest — replay the held-out Nov–Dec weeks like a diary and count what the top-20 list would have caught: 41 of 54 failures, 104 downtime days avoided, ≈ $2.66M net savings (~78% of the do-nothing downtime bill). Cross-checked: 41/54 ≈ the model's recall@20 of 0.78 — money story and metric story agree.

7

Then I proved it runs on the real stack.

Uploaded the data to Databricks (Free Edition) and ran the same PySpark feature job on a real cluster — identical answer, 24,867 × 226. Loaded the outputs into Snowflake (trial on Azure UK South) and rebuilt the analytics in pure SQL — including recomputing recall@20 in the warehouse. Details in the tabs below.

8

Module 2: a Keras autoencoder for the failures nobody labelled — with an honest result.

The classifier only knows failure patterns it has labels for. So I trained a Keras/TensorFlow autoencoder (196 → 64 → 16-number bottleneck → 64 → 196) on healthy weeks only, on Databricks serverless — machines drifting off "normal" reconstruct badly, and that reconstruction error is a label-free anomaly score. Result, reported straight: PR-AUC 0.111 — 9× random, beats IsolationForest (0.074), and loses to the simple per-sensor z-score alarm (0.202). I shipped the comparison instead of hiding it — see the Q&A for why that's the strongest card in the deck.

03 · The cloud part, explained like I'm 5

Why Databricks and Snowflake if my laptop already did it?

Short answer: my laptop proves the idea; the cloud proves it scales and survives me. A laptop can't hold a real fleet's data (250,000 machines, terabytes of logs), dies when I close the lid, and nobody else can query it. Click through:

Think of the data's journey as: warehouse-of-ingredients → factory → showroom.

StageToolLike-I'm-5
Storage (raw logs)Azure Blob / Parquet (we used local Parquet + a Databricks Volume)a giant cheap shed where every raw log file is kept forever, untouched
Processing (make features)Databricks (runs Spark)the factory: hundreds of workers chop the raw logs into the 226 features, in parallel

My laptop played all three roles at once — fine for 500 machines, impossible for 250,000. The cloud splits the roles so each part scales on its own, runs on a schedule without me, and is shared: the analyst in Germany and the service manager in India query the same Snowflake table. And this is precisely the stack in the Siemens JD — so I ran my pipeline on it for real rather than just naming the logos.

04 · Module 2 — the anomaly detector, start to finish

Catching the failures nobody labelled

The one-liner: Module 1's XGBoost is supervised — it only learns failure patterns we already labelled, and we only had about 440 known failures. Real fleets also break in ways nobody has annotated yet. So Module 2 is the safety net: an unsupervised anomaly detector that never sees a failure label. It learns what "normal" looks like for each machine, then flags when behaviour drifts away from that normal.

1

What we fed it — a fortnight of behaviour per machine.

Say it like this

"For every machine, every Monday, I take the last 14 days of telemetry across 14 sensors — a fortnight of behaviour. I z-score it per machine: each sensor is measured against that machine's own history, so a machine that's naturally hot doesn't look anomalous forever just because it runs hotter than the fleet. We're not asking 'is this high?' — we're asking 'is this high for this machine?' Extreme spikes get clipped, then I flatten the window into 196 numbers — 14 days × 14 sensors. Across the fleet that's 23,922 windows."

Jargon to keep: telemetry · z-score per machine · fortnight window · flatten · 196-dim input

2

The ML — an autoencoder, i.e. a photocopier.

Say it like this

"I used a Keras/TensorFlow autoencoder — the classic deep-learning tool for anomaly detection in predictive maintenance, and it hits the JD's Keras+TensorFlow ask where it actually fits. Think of it as a photocopier shaped 196 → 64 → 16 → 64 → 196: it has to squeeze a whole fortnight through a bottleneck of just 16 numbers, then rebuild the original 196. Target equals input, so the loss is MSE — how badly the copy differs from the original. I trained it only on healthy windows — 14,760 of them, no failure within 14 days — so it only ever learns 'normal.' Adam, batch 256, up to 40 epochs with early stopping, fixed seed so it reproduces."

Why it works: it can compress and rebuild normal behaviour cleanly; weird behaviour won't compress well, so the rebuild comes out blurry — and that blurriness is the signal.

Jargon to keep: autoencoder · bottleneck · reconstruction · MSE · trained on healthy only · unsupervised · Adam · early stopping

3

Where it trained — a Databricks war story.

Say it like this

"Deep-learning training belongs on the compute platform, not my laptop — so I ran it on Databricks serverless in databricks_autoencoder.ipynb, with the windows in the same fleetpulse volume. One real gotcha: %pip install tensorflow isn't enough — pip installs the library, but the running Python session still holds the old environment. You have to call dbutils.library.restartPython(), then train. Classic Databricks gotcha — we hit it and fixed it live."

Jargon to keep: Databricks serverless · volume · %pip install → restartPython()

4

Scoring — blurriness is the anomaly score.

Say it like this

"After training, every one of the 23,922 machine-weeks goes through the network. The reconstruction error — the mean squared difference between the input and the copy — is that week's anomaly score. Machines drifting off their own normal reconstruct badly and score high. The distributions show real separation — failing weeks sit to the right of healthy ones — but there's honest overlap, and about 15% of failures are sudden, so no telemetry-based detector can see those coming."

Jargon to keep: reconstruction error · anomaly score · separation with overlap

5

The verdict — say the numbers, then defend them.

Say it like this

"Same test window, same labels, same metric for everyone. Autoencoder PR-AUC 0.111 — about 9× random (0.013); it beats IsolationForest (0.074), but loses to the simple per-sensor z-score alarm (0.202), and it's far below supervised XGBoost (0.479). Why it lost: my score averages error over all 196 inputs, which dilutes a single-sensor drift — and single-sensor drift is this fleet's dominant failure signature, which the per-sensor rule catches directly. Roadmap: score by max per-sensor error, one model per modality, maybe an LSTM autoencoder for temporal shape. It keeps its place because it's the only detector that needs no labels — the net for failure modes nobody has annotated yet."

The interview flex: reporting that your deep model lost to a simple rule shows you care about the honest comparison, not the buzzword.

My 30-second spoken version (Module 2)

"Module 2 is my safety net. XGBoost only knows labelled failures — but machines also break in new ways. So I trained a Keras autoencoder on Databricks: it compresses a fortnight of telemetry through a 16-number bottleneck and rebuilds it, trained only on healthy weeks. When a machine drifts off its own normal, reconstruction error spikes — that's my label-free anomaly score. Honest result: PR-AUC 0.111, 9× random, beats IsolationForest, loses to a per-sensor z-score at 0.202 — because averaging over 196 dims dilutes single-sensor drift. Next I'd score max per-sensor error. The point isn't that deep learning always wins — it's knowing when it doesn't, and still having a net."

memorise cold: 0.111 / 0.013 / 0.074 / 0.202 / 0.479 — that's the credibility spine

If they poke

They askYou say
Why not just use XGBoost?XGBoost is supervised — it only catches failure modes I've already labelled, which is exactly Module 1's blind spot. Module 2 has to be unsupervised so it can flag failures nobody has labelled yet. Same tool would be the same blind spot.
Why z-score per machine?Machines have different baselines — a naturally hot scanner isn't broken. Per-machine z-score asks “high for this machine?” not “high for the fleet?”, stripping out each machine's personality so only genuine drift is left. Fleet-wide thresholds would punish it forever just for running hot.
Why train on healthy only?So it learns to copy only normal. If failing windows got in, it'd reconstruct sick machines cleanly too — and they'd stop standing out, defeating the detector. Healthy-only means anomalies come out as bad copies. (One-class learning.)
Why did the z-score win?The failure signature here is single-sensor drift, but my score averages error over all 196 sensors — one loud sensor diluted by 195 quiet ones. The per-sensor rule checks each alone, so it catches it directly. It won on fit, not smarts. Fix: score the max per-sensor error, not the mean.
Why keep Module 2 at all?It's the only label-free detector — the safety net for failure modes nobody has annotated yet. Not the best scorer today, but the only one that could flag a brand-new failure type — and 0.11 is a v1, not a ceiling (max-error, per-modality, LSTM all on the table).

04b · Module 3 — the engineer copilot, start to finish

"When this machine did this before — what fixed it?"

The one-liner: Module 1 predicts which machines will fail; Module 2 flags weird behaviour with no labels. Module 3 helps the engineer once they're standing at the machine: it takes the symptom and retrieves the most similar past service tickets, so the fix that worked last time is one search away. It's a retrieval assistant, not a chatbot — deliberately no LLM.

1

What we fed it — past tickets, symptom side only.

Say it like this

"I indexed 440 past corrective tickets, but only the symptom half — the complaint — never the resolution. That's deliberate: if I indexed the fix, I'd be leaking the very answer I'm trying to retrieve. So the search matches on how the problem looked, then surfaces what fixed it."

Jargon to keep: corrective tickets · symptom vs resolution · leakage-safe indexing

2

The ML — TF-IDF + cosine, and why not an LLM.

Say it like this

"I used TF-IDF with cosine similarity — turn each ticket's symptom text into a bag-of-words vector weighted so rare, informative words count more, then find the closest past tickets by cosine angle and show their resolutions. I chose it over an LLM on purpose: this is retrieval, not generation. It's explainable — I can point at the exact words that matched — it runs offline with no API key, and it can't hallucinate a fix that never happened."

Why not embeddings yet: TF-IDF is the right dependency-free v1. Embeddings are the roadmap — and I've got the measured evidence for when they'd help (below).

Jargon to keep: TF-IDF · cosine similarity · bag-of-words · retrieval not generation · explainable · offline

3

Where it runs — 187KB of JSON, scored in the browser.

Say it like this

"It ships as a 187KB JSON file the browser scores itself — no server, no API, no model to host. The catch is I have two scorers: the Python one in the pipeline and a TypeScript one in the browser, and they have to stay in lockstep — same stopword list, same tokenizer, same sublinear-TF, same L2 normalisation — or they'd rank the same query differently."

Jargon to keep: client-side scoring · Python/TS parity · stoplist · tokenizer · sublinear-TF · L2 norm

4

The honest numbers — and the trap in them.

Say it like this

"Leave-one-out scored a perfect 1.000 — which is worthless, and I say so: the tickets are templated, so a ticket just matches its near-identical twin. That's string-matching, not intelligence. The real test is held-out paraphrases: P@1 0.636, top-10 consensus 0.682, against a majority-class baseline of 0.452. So it genuinely beats guessing — but it's lexical, and I'm honest that leave-one-out flattered it."

Jargon to keep: leave-one-out (worthless here) · held-out paraphrases · P@1 0.636 · baseline 0.452

5

The bug war story — and the verdict.

Say it like this

"I found an evaluation bug worth retelling. My eval ranked all 440 tickets and took the top one — but a query with no overlapping words scores 0.0 against every ticket, and argsort on an all-zero array still returns ticket 0. So a total miss was being counted as a hit, inflating P@1 from the true 0.636 up to 0.682. I caught it by making the browser and the pipeline score the same 22 queries and demanding they agree — they now agree 22 out of 22. The rule I took from it: no match must mean no match. And every remaining miss is lexical, not conceptual — 'grinding noise' retrieves a detector ticket because they share the word 'noise' — which is exactly the measured argument for upgrading to embeddings later."

The interview flex: I didn't just build a search box — I stress-tested my own metric and found it was lying to me. That's the difference between a demo and production discipline.

My 30-second spoken version (Module 3)

"Module 3 is the engineer's copilot. When someone's at a machine, it takes the symptom and retrieves the most similar past tickets — so the fix that worked before is one search away. I built it with TF-IDF and cosine, not an LLM, on purpose: it's retrieval, it's explainable, it runs in the browser with no API, and it can't hallucinate. Honest numbers: on held-out paraphrases P@1 is 0.636 versus a 0.452 baseline — leave-one-out said 1.0 but that's worthless on templated tickets, and I say so. I also caught an eval bug where a total miss counted as a hit, by forcing my Python and browser scorers to agree on 22 queries. Every miss is lexical, not conceptual — which is my measured case for moving to embeddings next."

06 · What every number/chart means

The two-layer cheat sheet

Recall@20 = 0.78

In the interview: "Of all the failures that actually happened, 78% were already on our weekly 20-machine inspection list."
🔍 For my understanding: "@20" means "when only allowed to pick 20." 54 fires broke out; 41 were in buildings we'd flagged. Recall = did we find the bad ones. Its twin, precision@20 = 0.23, asks: of the 20 we visited, how many were really failing? ~1 in 4–5. We accept wasted $800 visits to avoid $54,000 surprises — the 100:1 cost asymmetry decides the trade-off, not vanity metrics.

PR-AUC = 0.48 (random ≈ 0.013)

In the interview: "Precision-recall area — the right headline metric at 1.7% prevalence; we're ~37× better than chance, and ~2× the best simple rule (a z-score alarm scores 0.20)."
🔍 For my understanding: accuracy fails on rare events (say "no failure" always → 98% accurate, catches nothing). PR-AUC only rewards you for finding the rare positives without crying wolf. The 0.48 isn't "48% correct" — it's a curve area; what matters is the comparison: random 0.013, simple rules 0.20, us 0.48.

Calibration (Brier 0.009)

In the interview: "Predicted probabilities match observed frequencies — of machines scored 40%, about 40% actually failed. That's what makes risk × cost arithmetic legitimate."
🔍 For my understanding: an uncalibrated model is like a friend who says "100% sure" about everything — directionally right, numerically useless. Platt scaling re-maps scores onto honest probabilities. We proved it in SQL too: bucket machines by predicted risk, compare to actual failure rate per bucket — the two columns track each other.

Autoencoder PR-AUC = 0.111 (Module 2)

In the interview: "My label-free anomaly detector: nine times better than random, better than IsolationForest — but it lost to the per-sensor z-score alarm at 0.202, and I report that openly. Its value is catching failure modes nobody has labelled; its next iteration is scoring by max per-sensor error instead of the mean."
🔍 For my understanding: why did it lose? The autoencoder averages its "blurriness" over all 196 numbers, so one slowly-drifting sensor gets drowned out by 195 well-copied ones — and one-sensor drift is exactly how this fleet usually fails. The z-score alarm checks each sensor separately, so it's hand-shaped for that signature. Lesson: a domain-fit simple rule beats a generic deep model — knowing when deep learning helps is the skill.

Net savings ≈ $2.66M (backtest)

In the interview: "Over the two held-out months, the weekly worklist would have avoided 104 downtime days: $2.81M avoided minus $144k of inspections — about a 78% cut in the downtime bill. Assumptions disclosed: $27k/down-day, $800/visit, an inspection converts a breakdown into a planned visit."
🔍 For my understanding: this is expected value logic made concrete. The number scales with the assumptions (synthetic data, upsampled failure rate) — so I lead with the percentage (~78%) which is robust, and quote the dollars as an illustration, never a promise.

08 · The conversation

They ask / I'd say

Your data is synthetic — why should I believe anything?

"You shouldn't believe the absolute numbers — I don't either; failure rates are deliberately upsampled so one year is trainable, and that's disclosed on every artifact. What the synthetic fleet buys is method proof: leakage-safe features, honest metrics at 1.7% prevalence, calibration, cost-based thresholds. With your real fleet, my first two weeks would be sitting with service engineers finding where my generator's assumptions are wrong — that's the fastest path to a trustworthy model."

Land this: honesty + a concrete first-two-weeks plan. Never defend the dollars.

Why XGBoost and not something newer?

"440 positive examples, tabular features, and users who need reasons. Boosted trees are still the strongest baseline on that shape of data, handle missing sensor values natively, and SHAP gives per-machine explanations a service planner can read. I benchmarked against simpler rules — z-score alarms, IsolationForest, age-ranking — and XGBoost roughly doubles the best of them. Newer isn't a reason; better-on-this-data is."

What exactly is leakage, and how do you know you don't have it?

"Leakage is when training data smuggles in information from the future — like using the failure-day error burst to 'predict' that day's failure. Three defences: every rolling window ends at the scoring day (rangeBetween(-(w-1), 0)); labels come from a strictly later window (t, t+7]; and a unit test fails the build if any feature touches post-scoring-day data. Also: time-based split, because random splits leak machine identity."

For me: this is the question that separates levels — have the three defences ready in order.

Why did you run it on Databricks when your laptop could handle it?

"Two reasons. Credibility: the JD names Databricks and Snowflake, so I wanted 'I ran my pipeline there' rather than 'I read about it'. Architecture: the code was written in PySpark precisely so scale is a config change, not a rewrite — and the cloud run proved it: identical 24,867 × 226 output. On your fleet the same job points at Blob storage and a bigger cluster; nothing else changes."

What's the difference between Databricks and Snowflake? Why both?

"Different jobs. Databricks is compute over the data lake — heavy transformation, ML, streaming; it's where raw logs become features. Snowflake is the analytical warehouse — clean tables, fast SQL, easy sharing and governance; it's where the business asks questions. Factory versus showroom. In our slice: 6 raw tables went to the factory, 4 finished tables to the showroom — the raw telemetry never needs to sit in the warehouse because the features already summarise it."

Your deep-learning model LOST to a one-line rule. Why show that?

"Because it's true, and because the comparison is the deliverable. The autoencoder scored 0.111 — well above random and IsolationForest — but the per-sensor z-score alarm scored 0.202, and supervised XGBoost 0.479. The reason is mechanical: my score averages reconstruction error over 196 inputs, which dilutes single-sensor drift — the dominant failure signature here — while the z-score rule checks sensors one by one. Next iteration: max per-sensor error, per-modality models, an LSTM autoencoder for temporal shape. But the honest ranking ships first. If I hid my model losing to a baseline, you shouldn't trust my $2.6M savings number either."

Land this: this answer converts a 'failure' into the strongest trust signal in the interview. Never rescue the number — sell the discipline.

What broke along the way? (they love this one)

"Plenty — and each one taught a portable lesson. Snowflake's wizard loaded my Parquet as a single VARIANT column, so I flattened it with typed views and learned its epoch-microsecond timestamp dance. rows turned out to be a reserved word. And I loaded the wrong file into a table during repeated uploads — caught it because the row count was 24,867 where 440 should have been. Rule burned in: verify row counts after every load, cross-check every metric two ways."

Land this: engineers trust people who tell debugging stories with specifics.

How would this go to production?

"Weekly batch: Data Factory (or Databricks Workflows) triggers the feature job Monday 6am, scores land in Snowflake, the dashboard reads them. Monitoring is designed but not built — PSI/KS drift on the feature distributions, performance tracked on outcomes as labels mature, a retrain trigger, and a human-in-the-loop rule: FleetPulse recommends inspections; it never auto-dispatches. In a medical context that last line is non-negotiable."

One thing worth knowing cold

Parquet — the answer to "why not CSV?": it stores data by column, not by row, so a query touching 3 of 226 columns reads ~1% of the file; it's compressed (~5–10× smaller), carries its own schema (types travel with the data — no guessing), and is splittable so Spark can chew one file with 100 workers in parallel. It's the lingua franca: pandas, Spark, Databricks, Snowflake, DuckDB all read it natively. One sentence version: "CSV is a text printout; Parquet is a database file — typed, compressed, columnar, and parallel-friendly."
