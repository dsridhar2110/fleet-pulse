01

The problem — downtime is the enemy

When a hospital scanner fails unexpectedly, everyone loses at once: the hospital cancels 20–40 patient scans, the manufacturer pays SLA penalties and dispatches an engineer reactively, and the service team — sitting on millions of log rows — finds out when the phone rings. The economics are lopsided: one day of unplanned downtime ≈ £25–30k; a proactive inspection visit ≈ £600. Missing a failure costs roughly 100× a wasted visit. FleetPulse's job is to turn the logs into three things a service organisation can act on: who to inspect this week, what's drifting off normal, and what fixed this fault last time.

The data — six tables, one fleet

Machine logs from a physics-based simulator: components wear out on Weibull lifetimes, degradation leaks into the sensors for weeks before most failures, ~15% of failures give no warning at all, and the data is deliberately imperfect — missing days, calibration offsets, engineers writing "He level low" one day and "cryo issue" the next. It's a star schema: one dimension table (the machines) and five event tables that join it on machine_id — in plain words, many events belong to one machine.

TableRowsOne row =Key
fleet_master500one machine (the asset register)PK machine_id
telemetry1,041,129one machine-day-sensor readingPK (machine_id, date, sensor)

Feature engineering rolls the events up to one row per machine-week — 24,867 rows × 226 features — with the 7-day failure label taken from failures only, a strict time-based split, and a unit test that no feature can see past its scoring day. Full deep-dive: reports/data-model-eda/.

Who uses it — three personas, one build

Same scores, three altitudes: a worklist for operations, an explanation for the engineer, a P&L line for the director.

P1 · Service Operations Manager

"Priya" — runs regional dispatch

"Which machines get my engineers this week?"

Opens the Command Center Monday 8am → KPI band → ranked worklist → filters her region → assigns the top risks. Cares about precision (wasted visits burn engineer-days) and the cost panel that justifies her calls.

P2 · Field Service Engineer

"David" — on the road

"I'm assigned FP-MRI-0103 — what am I walking into?"

Machine page → sees helium level falling 3 weeks → reads the top risk drivers in component language → asks the Copilot "have we seen this before?" → gets 3 similar past tickets + the part that fixed them → orders it before driving out.

P3 · Customer Service Director

"Elena" — owns the P&L and SLAs

"Is predictive maintenance actually saving us money?"

Executive view → fleet availability, failures caught early vs missed, cost avoided vs inspection spend, SLA trend. Cares about one number — net savings — and whether false alarms are eroding engineer trust.

06

The three modules

engine built · wiring this sprint

Problem statement: rank the fleet by probability of failure in the next 7 days, so a bounded weekly inspection budget catches the most failures — and price every decision in money.

Data in

All six tables → 226 features per machine-week: rolling sensor statistics and trends over 7/14/30-day windows, error-burst counts, maintenance recency, machine age and usage.

How it works

An XGBoost classifier with Platt calibration — calibration means a score of 0.4 really behaves like a 40% chance, so a planner can trust it. Chosen over deep learning deliberately: tabular data, few positives, native missing-value handling, and SHAP explainability. Honest results on the held-out test window: PR-AUC 0.48 (random would be 0.013), recall@20 = 0.78, median lead time 3 days. On top sits the Business Impact Engine — a deterministic money layer: expected loss per machine, cost avoided by the top-20 worklist, inspection spend, net savings. That's the panel P1 defends and P3 reports.

Problem statement: the classifier can only find failure patterns it has labels for. Catch machines behaving abnormally in ways we've never labelled — the unknown unknowns — and surface them as early warnings.

Honest result: test PR-AUC 0.111 — ~9× random (0.013), above IsolationForest (0.074), below the per-sensor z-score alarm (0.202) and supervised XGBoost (0.479). Reported openly; improvement path documented (max per-sensor error, per-modality models, LSTM autoencoder).

Data in

Telemetry only, no labels needed (unsupervised). Sliding windows of per-sensor daily readings, normalised per machine so a naturally-hot machine isn't permanently "anomalous".

How it works

A Keras/TensorFlow autoencoder — a neural network trained to compress and reconstruct normal telemetry. When reconstruction error spikes, the machine has left normal behaviour. It's benchmarked against the simpler baselines already in the repo (z-score, IsolationForest) — and if it can't beat them, we say so. Its scores feed the dashboard's "watch" tier between healthy and critical, and it closes the JD's explicit Keras/TensorFlow ask with the standard predictive-maintenance use of an autoencoder — not a bolted-on demo.

Problem statement: a field engineer's best asset is the memory of 49 colleagues' past repairs — locked in 1,610 free-text tickets written in inconsistent shorthand. Make that memory queryable in plain English.

Data in

Ticket notes, failure/repair history, the model's per-machine risk drivers, and small synthetic service-manual snippets. One guardrail worth saying out loud: ticket text is written after a failure, so it feeds retrieval — never the classifier's features. That would be label leakage.

How it works

The RAG pattern (Retrieval-Augmented Generation), retrieval-first: ticket text → embeddings → a local vector store (ChromaDB). Ask "why is FP-MRI-0103 high risk, and what fixed this before?" and the copilot pulls the machine's risk drivers and the most similar historical tickets, composing an answer with citations. The agentic part: it chooses which tools to call — risk lookup, ticket search, parts history — before answering. The deployed version is retrieval-only, so every answer traces to a real ticket; an Azure OpenAI generation layer is the documented plug-in point. Next step on the same data: the Neo4j knowledge graph (machine → component → failure → ticket → engineer) in the companion service-graph project.
